Quando objetos são semelhantes

Boas práticas de programação incluem não duplicar código. Uma das formas de reaproveitar código é através de herança.

Herança básica

Todas as classes em Python herdam implicitamente da classe especial object, que possui alguns métodos incorporados.

A classe herdada é chamada de superclasse ou pai. A classe que herda é chamada de subclasse ou filha. Herdar uma classe também é chamado de estender uma classe.

Basicamente, herança é usada para adicionar ou alterar funcionalidades em uma classe.

Vamos aplicar o conceito em um aplicativo gerenciador de contatos.

class Contact:
    all_contacts = []

    def __init__(self, name, email):
        self.name = name
        self.email = email
        Contact.all_contacts.append(self)

A variável all_contacts é uma variável de classe. Semelhante a uma variável global, há apenas UMA variável de classe Contact.all_contacts compartilhada por todo o sistema. Para acessar uma variável de classe, informe o nome da classe antes do nome da variável.

Se teu sistema possuir contatos "fornecedores", os quais poderão receber pedidos, não seria interessante implementar um método de pedidos diretamente na classe Contact, pois isso permitiria a contatos que não são fornecedores também receberem pedidos:

class Supplier(Contact):
        def order(self, order):
            print(f"If this were a real system we would send '{order}' order to '{self.name}'")

Tente instanciar objetos Contact e Supplier e chamar o método order para ver o que acontece.

Extendendo tipos incorporados

Podemos estender a funcionalidade de listas em Python para buscar por um item específico:

class ContactList(list):
    def search(self, name):
        """Return all contacts that contain the search value in their name."""
        matching_contacts = []
        for contact in self:
            if name in contact.name:
                matching_contacts.append(contact)
        return matching_contacts

Faça a seguinte alteração na classe Contact:

class Contact:
    all_contacts = ContactList() # altere aqui

    def __init__(self, name, email):
        self.name = name
        self.email = email
        Contact.all_contacts.append(self)

Lembrando, em Python, podemos inicializar listas com [], que é um atalho para list().

Outros tipos incorporados do Python são object, list, set, dict, file, int e float, que podem ser herdados para adicionar funcionalidades.

Sobrescrita e super

Além de adicionar funcionalidades, herança também pode alterar funcionalidades. Vamos sobrescrever o construtor de Contact:

class Friend(Contact):
        def __init__(self, name, email, phone):
            super().__init__(name, email) # chama o método __init__ pai
            self.phone = phone

A função super é usada dentro de uma classe para se referir à classe pai e não pode ser usada fora do escopo de uma classe. Ela acessa os atributos e métodos declarados na classe pai, útil quando sobrescrevemos um método na classe filha.

Herança múltipla

Herança múltipla envolve uma subclasse herdando de múltiplas superclasses, obtendo acesso às funcionalidades de cada uma. Embora o conceito pareça simples, muitas vezes é menos prático do que o esperado, levando muitos programadores experientes a desaconselhar o seu uso.

O mixin é a forma mais simples e útil de herança múltipla. Ele é uma superclasse destinada a ser herdada por outras classes para adicionar funcionalidade extra. Por exemplo, ao adicionar a capacidade de enviar emails à classe Contact, podemos criar um mixin que lida com o envio de emails:

class MailSender:
    def send_mail(self, message):
        print("Sending mail to " + self.email)
        # Add e-mail logic here

Embora o mixin por si só não tenha uma função especial (nem pode ser uma classe independente funcional), ele permite a criação de uma nova classe que combina características de Contact e MailSender, usando herança múltipla:

class EmailableContact(Contact, MailSender):
    pass

Vamos considerar alternativas ao mixin:

  • Herança simples: Poderíamos adicionar a função send_mail à subclasse. No entanto, isso resultaria em duplicação da funcionalidade de email para outras classes que também precisassem disso.
  • Função independente: Poderíamos criar uma função separada para enviar emails e chamá-la com o endereço de email correto quando necessário, evitando duplicação de código.
  • Composição: Em vez de herança, poderíamos usar composição, onde a classe EmailableContact teria um objeto MailSender como atributo.
  • Monkey patching: Poderíamos adicionar um método send_mail à classe Contact após sua criação, recurso esse que veremos mais tarde.

Herança múltipla é eficaz para combinar métodos de diferentes classes, mas se complica ao lidar com chamadas de métodos de mesmo nome em várias superclasses.

Para explorar essas questões, consideremos a adição de um endereço residencial à classe Friend. Existem várias abordagens possíveis: passar os elementos do endereço como parâmetros para o construtor da classe Friend, armazená-los em uma estrutura (como uma tupla ou dicionário) e passá-los como um único argumento ou até mesmo criar uma classe separada chamada Address para encapsular esses elementos e passar uma instância dela ao construtor da classe Friend.

Embora a composição seja uma solução eficaz para esse cenário, a herança também é possível. Neste contexto, criamos uma nova classe chamada AddressHolder:

class AddressHolder:
    def __init__(self, street, city, state, code):
        self.street = street
        self.city = city
        self.state = state
        self.code = code

Esta classe simplesmente armazena os elementos do endereço como variáveis de instância durante a inicialização.

O problema do diamante

Alterar Friend para herdar de AddressHolder gera ambiguidade ao chamar o método super().__init__. De qual superclasse estaríamos nos referindo? Poderíamos resolver da seguinte forma:

class Friend(Contact, AddressHolder):
    def __init__(
        self, name, email, phone, street, city, state, code):
        Contact.__init__(self, name, email) # chame o construtor do pai Contact
        AddressHolder.__init__(self, street, city, state, code) # chame o construtor do pai AddressHolder
        self.phone = phone

O problema acontece quando ambas as superclasses também herdam de uma classe em comum, formando um diamante (ou losango) no diagrama de classes:

Nesse exemplo, como todas as classes herdam de object, o método __init__ de object seria chamado duas vezes. Como este não faz nada, aqui, não teria problema. Mas imagine tentar se conectar ao banco de dados duas vezes para cada requisição! Portanto, a classe base deveria ser chamada apenas uma vez. Mas por quem? Em qual ordem? Friend → Contact → Object?

Considere mais um exemplo do problema do diamante:

Vamos codigar esse diagrama:

class BaseClass:
    num_base_calls = 0

    def call_me(self):
        print("Calling method on Base Class")
        self.num_base_calls += 1


class LeftSubclass(BaseClass):
    num_left_calls = 0

    def call_me(self):
        BaseClass.call_me(self)
        print("Calling method on Left Subclass")
        self.num_left_calls += 1


class RightSubclass(BaseClass):
    num_right_calls = 0

    def call_me(self):
        BaseClass.call_me(self)
        print("Calling method on Right Subclass")
        self.num_right_calls += 1


class Subclass(LeftSubclass, RightSubclass):
    num_sub_calls = 0

    def call_me(self):
        LeftSubclass.call_me(self)
        RightSubclass.call_me(self)
        print("Calling method on Subclass")
        self.num_sub_calls += 1

Instancie um objeto de Subclass e chame seu método call_me. Observe que call_me da classe base é chamado duas vezes. Isso pode ser um problema. Imagine um aplicativo de banco realizando o depósito duas vezes!

Substitua as chamadas explícitas aos métodos call_me das superclasses por apenas uma chamada super().call_me(). Assim, o método na classe base é chamado apenas uma vez.

Quando Subclass.call_me é executado, ele chama super().call_me, que pode ser uma das duas classes. Conforme a saída, foi chamado LeftSubclass.call_me. Este, por sua vez, ao executar super().call_me, ao invés de chamar o método pai, chama o próximo método, que é RightSubclass.call_me. Aí sim, este chama o método BaseClass.call_me. Portanto, as chamadas a super() garantem que cada método da hierarquia seja chamado apenas uma vez.

Diferentes conjuntos de argumentos

Chamar super().call_me deu certo no último exemplo pois os métodos esperavam o mesmo número de parâmetros. Mas os métodos __init__ das superclasses de Friend esperam receber uma quantidade diferente de parâmetros.

O problema de chamar métodos de superclasse com nomes iguais, mas conjuntos diferentes de argumentos, surge ao realizar herança múltipla.

A solução para esse problema é aceitar keyword arguments (kwargs). kwargs é uma forma de passar uma quantidade variada de argumentos ao método que não foram especificados na declaração do método. Assim, esse método pode repassar os argumentos não utilizados ao próximo na chamada a super().

class Contact:
    all_contacts = []

    def __init__(self, name="", email="", **kwargs):
        super().__init__(**kwargs)
        self.name = name
        self.email = email
        self.all_contacts.append(self)


class AddressHolder:
    def __init__(self, street="", city="", state="", code="", **kwargs):
        super().__init__(**kwargs)
        self.street = street
        self.city = city
        self.state = state
        self.code = code


class Friend(Contact, AddressHolder):
    def __init__(self, phone="", **kwargs):
        super().__init__(**kwargs)
        self.phone = phone

A sintaxe **kwargs coleta argumentos não definidos na lista de parâmetros do método, armazenando-os em um dicionário convencionalmente chamado kwargs. Ao chamar outros métodos passando kwargs como argumento, este é desempacotado, e os argumentos que coincidem com os parâmetros do método são removidos de kwargs. Abordaremos kwargs com mais detalhes no Capítulo 7. Para saber mais a respeito, acesse aqui.

Embora tenhamos resolvido o problema, o código não fica muito claro. Em casos onde não podemos fugir disso, docstrings podem ajudar a esclarecer quais argumentos são esperados pelo método.

Ao passar a variável **kwargs para super, a variável não inclui nenhum dos outros parâmetros. Isso pode causar problemas quando outras classes necessitarem desses parâmetros. Para garantir que uma variável seja transmitida para as classes superiores, existem diversas abordagens. Suponha que tanto a classe Contact quanto Friend precisem ser inicializadas com um parâmetro phone. Algumas opções são:

  1. Deixar a variável no dicionário kwargs, em vez de incluí-la explicitamente como parâmetro. As classes podem acessá-la usando kwargs['phone'].
  2. Tornar a variável um parâmetro explícito, mas atualizar o dicionário kwargs antes de passá-lo para super usando kwargs['phone'] = phone.
  3. Tornar a variável um parâmetro explícito, mas atualizar o dicionário kwargs usando o método kwargs.update, que é útil para adicionar vários argumentos de uma vez.
  4. Passar a variável explicitamente usando super().__init__(phone=phone, **kwargs).

kwargs é um recurso poderoso do Python, mas lembre-se da boa prática de programação: explícito é melhor que implícito.

A herança múltipla básica pode ser útil, mas em muitos casos é mais vantajoso optar por uma abordagem mais transparente, como a composição ou a aplicação de padrões de design abordados nos capítulos 10 e 11.

Polimorfismo

Definição: chamar um método idêntico para diferentes subclasses mas com comportamentos diferentes, sem explicitamente saber qual classe foi chamada. Por exemplo, um objeto da classe AudioFile pode ter um método play que se encarrega de descomprimir o arquivo de áudio e direcioná-lo para a placa de som. Mas dependendo do tipo de áudio, o processo de descompressão é diferente. Assim, poderíamos ter diferentes subclasses com o mesmo método implementado de forma diferente para diferentes arquivos de áudio, como mp3, wav, ogg. Um objeto “tocador de música”, então, não precisaria saber a subclasse de um objeto AudioFile, apenas que ele possui um método play que cumprirá seu papel quando chamado.

Exemplo:

class AudioFile:
    def __init__(self, filename):
        if not filename.endswith(self.ext):
            raise Exception("Invalid file format")

        self.filename = filename


class MP3File(AudioFile):
    ext = "mp3"

    def play(self):
        print(f"playing {self.filename} as mp3")


class WavFile(AudioFile):
    ext = "wav"

    def play(self):
        print(f"playing {self.filename} as wav")


class OggFile(AudioFile):
    ext = "ogg"

    def play(self):
        print(f"playing {self.filename} as ogg")

O método __init__ da superclasse consegue acessar a variável ext de diferentes subclasses. Polimorfismo nos permite isso.

Tente criar um script “tocador de música”, instanciar alguns objetos, inclusive passando um formato de arquivo inválido para um deles.

Duck typing

Como em Python as verificações de tipo geralmente são feitas em tempo de execução ao invés de tempo de compilação, além do polimorfismo hereditário, também podemos usar o recurso “duck typing” do Python.

A expressão mais comum que ilustra o conceito de duck typing é: "Se parece um pato, nada como um pato e grasna como um pato, então é provavelmente um pato". Isso significa que, em vez de verificar se um objeto é de uma classe específica, você verifica se ele possui os métodos e atributos esperados para executar uma determinada tarefa.

O duck typing nos permite criar objetos que funcionem de maneira semelhante aos tipos esperados, sem nos preocupar com herança ou implementação explícita de interfaces. Por exemplo:

class FlacFile:
    def __init__(self, filename):
        if not filename.endswith(".flac"):
            raise Exception("Invalid file format")

        self.filename = filename

    def play(self):
        print(f"playing {self.filename} as flac")

Apesar de FlacFile não herdar diretamente de AudioFile, podemos usá-la como se fosse, pois implementa a mesma interface:

def play(objeto):
    objeto.play()


mp3file = MP3File("file.mp3")
flacfile = FlacFile("file.flac")

play(mp3file)
play(flacfile)

Métodos de classe

Além dos métodos estáticos e métodos de instância, o Python também tem os métodos de classe. Os Métodos de classe são métodos que operam na classe em si, em vez de em instâncias individuais. Eles são decorados com @classmethod e seu primeiro argumento se refere à própria classe em que o método foi chamado, e convencionalmente o declaramos como cls.

Os Métodos de classe têm sua aplicação em situações mais complexas, como quando precisamos criar objetos de uma classe com diferentes configurações de maneira flexível, usando o padrão de design chamado "fábrica." Além disso, eles são utilizados em conceitos avançados de herança, que citaremos posteriormente. No entanto, em tarefas do dia a dia, geralmente não são essenciais e podem ser substituídos por alternativas mais simples, como métodos estáticos.

O seguinte código é um exemplo do padrão factory, adaptado daqui:

class Pizza:
    def __init__(self, ingredientes):
        self.ingredientes = ingredientes

    @classmethod
    def marguerita(cls):
        return cls(["molho de tomate", "manjericão", "rodelas de tomate fresco", "azeitona", "queijo muçarela", "orégano salpicado"])

    @classmethod
    def calabresa(cls):
        return cls(["queijo", "molho de tomate", "calabresa em rodelas", "cebola", "tomate picado", "azeite", "orégano"])


p1 = Pizza(["queijo", "azeitona preta", "ovo cozido", "presunto cozido", "cebola", "ervilha", "molho de tomate", "azeite"])
p2 = Pizza.marguerita()
p3 = Pizza.calabresa()

print("\nPizza de Portuguesa:", p1.ingredientes)
print("\nPizza de Marguerita:", p2.ingredientes)
print("\nPizza de Calabresa:", p3.ingredientes)

No método de classe, em vez de receber uma instância como o parâmetro self, ele recebe a própria classe como o parâmetro cls. Isso nos permite chamar o construtor da classe que foi passada como parâmetro. Você pode estar se perguntando por que não chamamos diretamente o construtor da classe Pizza em vez de passar cls como parâmetro. De fato, neste exemplo específico, poderíamos fazer isso e marcar o método como @staticmethod. No entanto, em situações em que a classe Pizza é uma superclasse, podemos não querer chamar diretamente o construtor da Pizza, mas sim ter acesso à classe filha, por exemplo, Calzone, que pode ter sido passada como parâmetro. Portanto, de fato, métodos de classe têm mais utilidade no contexto de herança.

Classes bases abstratas

Em Python, uma Classe Base Abstrata (ABC) define um conjunto de métodos e propriedades que subclasses são obrigadas a implementar.

Usando uma classe base abstrata

Python organiza sua API usando classes abstratas. Um exemplo delas é a classe Container, que possui o método abstrato __contains__(self, x), que verifica se x está contido no objeto desta classe. Este método é implementado pelas classes list, str e dict. Poderíamos criar um contêiner simplório que verifica se um parâmetro está contido no conjunto dos números inteiros pares:

class OddContainer:
    def __contains__(self, x):
        return isinstance(x, int) and x % 2 == 0

A função isinstance verifica se o primeiro argumento é uma instância do segundo argumento.

Mesmo não herdando de Container, OddContainer é considerado pelo Python como Container:

>>> from collections import Container
>>> odd_container = OddContainer()
>>> isinstance(odd_container, Container)
True
>>> issubclass(OddContainer, Container)
True

A função issubclass verifica se o primeiro argumento é uma subclasse do segundo argumento.

Isso é um exemplo de duck typing, podemos alcançar polimorfismo sem uso de herança.

Quando utilizamos o operador in no Python, ele chama implicitamente o método __contains__ do operando à direita passando o operando à esquerda como parâmetro:

>>> 1 in odd_container
True
>>> 2 in odd_container
False
>>> 3 in odd_container
True
>>> "a string" in odd_container
False

Criando uma classe abstrata

Uma das principais vantagens das classes abstratas é a facilitação na criação de APIs. A classe abstrata estabelece um contrato, ou seja, define quais métodos devem ser implementados pelas subclasses. Essa especificação da API é essencial para que os desenvolvedores entendam como interagir com as classes baseadas na abstração. Além disso, a própria classe abstrata, juntamente com comentários e descrições adequadas, compõe uma parte valiosa da documentação da API. Isso permite que os desenvolvedores compreendam a estrutura da classe abstrata, seus métodos e propriedades, além de entenderem como criar novas classes que se encaixem nesse padrão. Seguindo a API você e outros desenvolvedores podem desenvolver diferentes versões de classes abstratas para diferentes propósitos, e outros desenvolvedores poderão usar as diferentes versões de igual modo. Um exemplo comum de APIs é em bancos de dados, tendo uma API em comum você pode implementá-la para um banco de dados específico, e outras pessoas que utilizam dessa API podem trocar para o seu banco de dados simplesmente mudando para a tua implementação sem necessitar modificar o restante do sistema.

Vamos ilustrar ABC com o exemplo de mídia de áudio:

import abc


class MediaLoader(abc.ABC):
    @abc.abstractmethod
    def play(self):
        pass

    @property
    @abc.abstractmethod
    def ext(self):
        pass


class Ogg(MediaLoader):
    ext = ".ogg"

    def play(self):
        pass


o = Ogg()
print(o)
print(issubclass(Ogg, MediaLoader))
print(isinstance(o, MediaLoader))


class Mp3:
    ext = ".mp3"

    def play(self):
        pass


m = Mp3()
print(m)
print(issubclass(Mp3, MediaLoader))
print(isinstance(m, MediaLoader))


class Wav(MediaLoader):
    pass


w = Wav()
print(w)
print(issubclass(Wav, MediaLoader))
print(isinstance(w, MediaLoader))

Para uma classe ser abstrata, deve herdar de ABC. Uma classe abstrata pode ter métodos concretos. Para marcar um método como abstrato, decore-o com @abc.abstractmethod. O que tiver essa decoração deve ser implementado na classe filha. Já a decoração @property, que não tem a ver com ABC, marca um método como propriedade, isto é, ele é considerado como uma variável somente-leitura. Veremos mais sobre decorações no capítulo 10.

Observe que Ogg é uma subclasse de MediaLoader, mas Mp3 não. Quando você herda de uma classe base, o Python verifica automaticamente se a nova classe é uma subclasse válida da classe base. No entanto, em algumas situações, você pode querer personalizar ou relaxar essas verificações de subclasse através de duck typing. Para tanto, acrescente esse método à MediaLoader:

    @classmethod
    def __subclasshook__(cls, Subclass):
    if cls is MediaLoader:
        if hasattr(Subclass, "play") and hasattr(Subclass, "ext"):
            return True

    return NotImplemented

O método __subclasshook__ é frequentemente usado em classes base abstratas (ABCs) e pode ser usado para controlar se uma determinada classe é considerada uma subclasse da classe que define o método __subclasshook__. Ele deve ser definido como um método de classe (usando a decoração @classmethod) em uma classe base. O método recebe dois argumentos: a classe sendo verificada e a classe potencialmente considerada uma subclasse. Esse método é chamado automaticamente durante a execução das funções issubclass e isinstance, antes da verificação de herança.

Se o método retornar True, o Python considerará a classe como uma subclasse válida da classe base. Se o método retornar False, o Python não considerará a classe como uma subclasse válida da classe base. Se o método retornar NotImplemented, o Python continuará a verificação de subclasse olhando para as superclasses.

O uso de __subclasshook__ é mais avançado e geralmente não é necessário para a maioria das situações. Ele é mais útil em cenários onde você deseja customizar a verificação de subclasse para casos específicos ou quando lida com estruturas de herança complexas.

Tipos Enum

Um Enum é um conjunto de nomes simbólicos vinculados a valores únicos. Eles são semelhantes a variáveis globais, mas oferecem um método repr mais útil, agrupamento, segurança de tipo (type-safety) e alguns outros recursos.

Eles são úteis quando você precisa representar um conjunto fixo de constantes. Por exemplo, os dias da semana:

from enum import Enum
class Weekday(Enum):
    MONDAY = 1
    TUESDAY = 2
    WEDNESDAY = 3
    THURSDAY = 4
    FRIDAY = 5
    SATURDAY = 6
    SUNDAY = 7

Ou talvez as cores primárias RGB:

from enum import Enum
class Color(Enum):
    RED = 1
    GREEN = 2
    BLUE = 3

Como você pode ver, criar um Enum é tão simples quanto escrever uma classe que herda do próprio Enum.

Dependendo da natureza do enum, o valor de um membro pode ou não ser importante, mas de qualquer maneira esse valor pode ser usado para obter o membro correspondente:

>>> Weekday(3)
<Weekday.WEDNESDAY: 3>

Os membros do Enum têm um atributo que contém apenas o nome deles:

>>> print(Weekday.TUESDAY.name)
TUESDAY

Da mesma forma, eles têm um atributo para o valor deles:

>>> Weekday.WEDNESDAY.value
3

Se você tiver um membro enum e precisar do nome ou valor dele:

>>> member = Color.RED
>>> member.name
'RED'
>>> member.value
1

Membros e atributos permitidos de Enums

Enums são classes Python e podem ter métodos e atributos. Por exemplo, o seguinte Enum tem todos os planetas do nosso sistema solar, cada um com dois atributos associados:

from enum import Enum


class Planet(Enum):
    MERCURY = (3.303e23, 2.4397e6)
    VENUS = (4.869e24, 6.0518e6)
    EARTH = (5.976e24, 6.37814e6)
    MARS = (6.421e23, 3.3972e6)
    JUPITER = (1.9e27, 7.1492e7)
    SATURN = (5.688e26, 6.0268e7)
    URANUS = (8.686e25, 2.5559e7)
    NEPTUNE = (1.024e26, 2.4746e7)

    def __init__(self, mass, radius):
        self.mass = mass  # in kilograms
        self.radius = radius  # in meters

    def surface_gravity(self):
        # universal gravitational constant  (m3 kg-1 s-2)
        G = 6.67300e-11
        return G * self.mass / (self.radius * self.radius)

    def surface_weight(self, otherMass):
        return otherMass * self.surface_gravity()

Se o método __init__ for definido, o valor do membro enum será passado para ele.

O código a seguir pega seu peso na Terra (em qualquer unidade) e imprime seu peso em todos os planetas (na mesma unidade):

earth_weight = float(input("What is your weight on Earth? "))
mass = earth_weight / Planet.EARTH.surface_gravity()
for p in Planet:
    print(f"Your weight on {p.name} is {p.surface_weight(mass)}")

A maioria dos casos usa inteiros para valores de enum. Usar inteiros é curto e prático (e fornecido por padrão pela API Funcional), mas não estritamente forçado. Na grande maioria dos casos de uso, não importa o valor de um enum. Mas se o valor for importante, os enums podem ter valores arbitrários:

class Mood(Enum):
    FUNKY = 1
    HAPPY = 3

    def describe(self):
        # self is the member here
        return self.name, self.value

    def __str__(self):
        return f'my custom str! {self.value}'

    @classmethod
    def favorite_mood(cls):
        # cls here is the enumeration
        return cls.HAPPY

Então:

>>> Mood.favorite_mood()
<Mood.HAPPY: 3>
>>> Mood.HAPPY.describe()
('HAPPY', 3)
>>> str(Mood.FUNKY)
'my custom str! 1'

Como outro exemplo, datetime.date tem dois métodos para retornar o dia da semana: weekday e isoweekday. A diferença é que um deles conta de 0-6 e o outro de 1-7. Em vez de nos preocuparmos com isso manualmente, podemos adicionar um método ao enum Weekday para extrair o dia da instância de data e retornar o membro enum correspondente:

class Weekday(Enum):
    MONDAY = 1
    TUESDAY = 2
    WEDNESDAY = 3
    THURSDAY = 4
    FRIDAY = 5
    SATURDAY = 6
    SUNDAY = 7

    @classmethod
    def from_date(cls, date):
        return cls(date.isoweekday())

Agora podemos descobrir que dia é hoje! Observe:

>>> from datetime import date
>>> Weekday.from_date(date.today())
<Weekday.TUESDAY: 2>

Este enum Weekday é ótimo se nossa variável precisar apenas de um dia, mas e se precisarmos de vários? Talvez estejamos escrevendo uma função para planejar tarefas durante uma semana e não queremos usar uma lista...

Flag Enum

Ao criar um enum que herda de Flag, podemos executar operações bit a bit em seus membros, como combiná-los usando o operador |:

from enum import Flag

class Weekday(Flag):
    MONDAY = 1
    TUESDAY = 2
    WEDNESDAY = 4
    THURSDAY = 8
    FRIDAY = 16
    SATURDAY = 32
    SUNDAY = 64

Mudamos duas coisas: herdamos de Flag, e os valores são todos potências de 2.

Assim como o enum original Weekday acima, podemos ter uma única seleção:

>>> first_week_day = Weekday.MONDAY
>>> first_week_day
<Weekday.MONDAY: 1>

Mas o Flag também nos permite combinar vários membros em uma única variável:

>>> weekend = Weekday.SATURDAY | Weekday.SUNDAY
>>> weekend
<Weekday.SATURDAY|SUNDAY: 96>

Nos casos em que os valores reais dos membros não importam, você pode economizar algum trabalho e usar o método auto para os valores:

from enum import auto
class Weekday(Flag):
    MONDAY = auto()
    TUESDAY = auto()
    WEDNESDAY = auto()
    THURSDAY = auto()
    FRIDAY = auto()
    SATURDAY = auto()
    SUNDAY = auto()
    WEEKEND = SATURDAY | SUNDAY

Considere este código:

from enum import Flag
from enum import auto


class Color(Flag):
    BLACK = 0
    RED = auto()
    GREEN = auto()
    BLUE = auto()
    PURPLE = RED | BLUE
    WHITE = RED | GREEN | BLUE

Iterar sobre uma combinação de membros Flag retornará apenas os membros que são compostos por um único bit:

>>> Color(5)  # named combination
<Color.PURPLE: 5>
>>> Color(3)      # not named combination
<Color.GREEN|RED: 3>

Nomes de pseudo-flags são construídos a partir dos nomes de seus membros:

>>> (Color.RED | Color.BLUE).name
'PURPLE'

Flags de valor zero são sempre consideradas contidas:

>>> Color.BLACK in Color.WHITE
True

Caso contrário, será retornado True apenas se todos os bits de uma flag estiverem na outra flag:

>>> Color.PURPLE in Color.WHITE
True
>>> Color.GREEN in Color.PURPLE
False

    1. PDFs (autor, data de criação, tamanho do arquivo).

    4. Vídeos (autor, data de criação, tamanho do arquivo, dimensões da imagem, duração).

    Cada documento deve possuir um método que recebe uma unidade de medida de bytes (B, K, M, G, T, ...) e retorna seu tamanho nessa unidade.

    Se o documento tiver dimensões, também deve ter um método que retorna a quantidade de pixels.

    Como você implementaria esse sistema usando herança?

  2. Elabore o diagrama de classes e implemente em Python, utilizando herança e composição, a descrição a seguir. Considere Polígono e Quadrilátero como classes abstratas. Não é necessário realizar validação dos dados; suponha que os valores de inicialização dos objetos serão sempre válidos.

    Descrição:

    Um ponto é definido pelas coordenadas $x$ e $y$. É possível calcular a distância entre dois pontos.

    Uma linha é composta por dois pontos. É possível calcular a medida da linha, dada pela fórmula $d = \sqrt{(x_2 - x_1)^2 + (y_2 - y_1)^2}$. É possível calcular o coeficiente angular da linha, dado pela fórmula $m=\frac{y_2-y_1}{x_2-x_1}$, mas quando o denominador é zero, não há coeficiente angular. É possível verificar se duas linhas são paralelas quando seus coeficientes angulares forem iguais.

    Um polígono é composto por várias linhas conectadas que não se cruzam, chamadas de lados. É possível calcular a área e o perímetro de um polígono.

    Um triângulo é um polígono com três lados. Sejam $a$, $b$ e $c$ as medidas dos lados do triângulo. Seu semiperímetro é dado pela fórmula $p=\frac{a+b+c}{2}$. Sua área é dada pela fórmula $A=\sqrt{p(p-a)(p-b)(p-c)}$.

    Um quadrilátero é um polígono com quatro lados.

    Um quadrado é um quadrilátero com quatro lados de igual medida e todos os ângulos medindo 90°.

    Um trapézio é um quadrilátero com apenas duas linhas paralelas entre si, chamadas de base maior e base menor. Sua altura é a distância perpendicular entre suas duas bases, dada pela fórmula $$ h = \frac{ \sqrt{(-a + b + c + d)(a - b + c + d)(a - b - c + d)(a - b + c - d)} } {2\vert a - b \vert}$$, onde $a$, $b$, $c$ e $d$ representam as medidas das duas bases e dos dois lados oblíquos, respectivamente. Sua área é dada pela fórmula $A=\frac{a+b}{2}\cdot h$.
